 Syncscore — FastAPI Badge-Based Onboarding Backend

 Context

 Syncscore is a trust layer marketplace connecting AI service producers with non-technical buyers.
 The backend's core job is to implement a progressive 4-badge onboarding system that turns a raw
 sign-up into a publicly searchable, credibility-verified producer profile.
 Three subsystems drive the product's value: (1) badge state machine, (2) async artifact validation
 pipeline, (3) confidence scoring + LLM capability document generation.

 Phase 1 — ORM Models (SQLAlchemy 2.0 Mapped syntax throughout)

 app/models/user.py — User
 id (UUID PK), email (unique), hashed_password (nullable — OAuth users),
 entity_type (enum: individual/company), is_active (False until OTP verified),
 badge_1..badge_4 (bool), is_publicly_searchable (bool),
 oauth_provider (enum: google/linkedin, nullable), oauth_id (nullable),
 linkedin_url, github_url, created_at, updated_at
 Relationships: proof_artifacts (list), capability_declaration (one), capability_document (one)

 app/models/proof_artifact.py — ProofArtifact
 id (UUID PK), user_id (FK), artifact_type (enum: github_repo/live_demo/product_page/
 tech_blog/sandbox_deployment/case_study), url, validation_status (enum:
 pending/in_progress/passed/failed), validation_details (JSON), created_at
 JSON blob shape for github_repo: {accessible, last_push_date, recency_days, stars, commits_count, language, coherence_note}
 JSON blob shape for others: {accessible, http_status, recency_days, coherence_note}

 app/models/capability.py — CapabilityDeclaration + CapabilityDocument

 CapabilityDeclaration:
 id, user_id (FK unique), rag_capability+detail, agent_systems+detail,
 tool_orchestration+detail, memory_systems+detail, deployment_type
 (enum: cloud/on_prem/hybrid), frameworks (JSON array), created_at, updated_at

 CapabilityDocument:
 id, user_id (FK unique), declaration_id (FK),
 # Producer-editable:
 narrative (Text), positioning (Text), focus_industries (JSON array),
 # System-controlled (never writable via PATCH):
 verified_capabilities (JSON), declared_capabilities (JSON),
 confidence_score (int 0–100), tier (enum: Bronze/Silver/Gold),
 execution_verified (bool), raw_llm_output (Text), created_at, updated_at

 Alembic: alembic init alembic → configure env.py to import Base.metadata → alembic revision --autogenerate -m "initial" → alembic upgrade head

 Phase 4 — Services

 otp_service.py
 - generate_and_store_otp(email) → secrets.randbelow(1_000_000) zero-padded to 6 digits →
 bcrypt hash → redis.setex(f"otp:{email}", 600, hash) → returns plaintext OTP
 - verify_otp(email, otp) → get hash from Redis → pwd_context.verify → delete key on match (single-use)

 oauth_service.py (pure httpx, no OAuth library)
 - get_google_authorization_url() → generate CSRF state → store in Redis 10min → return (url, state)
 - exchange_google_code(code, state) → verify state → POST token exchange → GET userinfo → return {id, email, name}
 - Same pattern for LinkedIn (uses /v2/me + /v2/emailAddress)

 artifact_service.py (called via FastAPI BackgroundTasks)
 - validate_artifact(artifact_id, db):
   a. Set status=in_progress
   b. httpx.AsyncClient HEAD request → check 200, follow redirects
   c. If github_repo type → delegate to github_service.get_repo_signals(url)
   d. Build validation_details JSON
   e. Set status=passed/failed
   f. Call _check_badge_2_eligibility(user_id, db) — if all artifacts passed + profiles present → set badge_2=True + is_publicly_searchable=True

 github_service.py
 - get_repo_signals(github_url) → parse owner/repo → GET api.github.com/repos/{owner}/{repo} →
 return {stars, last_push_date, recency_days, commits_count_estimate, language}
 - Uses Authorization: token {GITHUB_TOKEN} header if configured

 confidence_engine.py
 Weighted scoring algorithm (total = 100):

 ┌──────────────────────────────┬────────┬──────────────────────────────────────────────────────┐
 │            Signal            │ Weight │                  Verified Condition                  │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ RAG capability               │ 15     │ github_repo artifact contains RAG keywords in README │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ Agent systems                │ 15     │ github_repo has relevant deps                        │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ Tool orchestration           │ 10     │ declared + artifact present                          │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ Memory systems               │ 10     │ declared + artifact present                          │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ GitHub activity (stars ≥ 10) │ 10     │ from github_service signals                          │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ GitHub recency (< 180 days)  │ 10     │ from github_service signals                          │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ Multiple artifacts (≥ 2)     │ 10     │ count of passed artifacts                            │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ LinkedIn verified            │ 10     │ profile URL accessible                               │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ Deployment type declared     │ 5      │ any value present                                    │
 ├──────────────────────────────┼────────┼──────────────────────────────────────────────────────┤
 │ Frameworks declared (≥ 2)    │ 5      │ len(frameworks) >= 2                                 │
 └──────────────────────────────┴────────┴──────────────────────────────────────────────────────┘

 Tier assignment: Bronze (0–39), Silver (40–69), Gold (70–100)

 compute_score(user_id, declaration, artifacts) → returns {score, tier, verified_capabilities, declared_capabilities}

 capability_doc.py
 - generate_capability_document(user, declaration, score_result) → calls Claude API
 - Prompt structure:
 System: You are Syncscore's capability document generator. Generate a structured
 professional profile for an AI service producer based on verified signals.

 User: Producer details:
 - Entity type: {entity_type}
 - Declared capabilities: {...}
 - Verified capabilities: {verified_capabilities}
 - Confidence score: {score}/100, Tier: {tier}
 - Proof artifacts: {artifact_types and urls}
 - Frameworks: {frameworks}

 Generate a JSON response with keys:
 "narrative" (2-3 paragraph professional description),
 "positioning" (1 sentence positioning statement),
 "focus_industries" (list of 3-5 relevant industries)
 - Parse JSON from response → populate CapabilityDocument fields
 - Store raw_llm_output for auditability

 Phase 5 — API Endpoints

 api/v1/auth.py
 POST /auth/register              → create user (inactive), generate+send OTP
 POST /auth/verify-otp            → verify OTP → set is_active=True, badge_1=True → return tokens
 GET  /auth/oauth/{provider}      → return {authorization_url}
 GET  /auth/oauth/{provider}/callback → exchange code → upsert user → return tokens
 POST /auth/login                 → verify password → return tokens
 POST /auth/token/refresh         → validate refresh token → return new access token

 api/v1/presence.py (requires badge_1)
 POST /presence/profiles          → save linkedin_url + github_url on User
 POST /presence/artifacts         → create ProofArtifact (status=pending) → enqueue background validation
 GET  /presence/artifacts         → list all artifacts for current user
 GET  /presence/artifacts/{id}/status → poll validation status
 POST /presence/verify            → manually re-trigger Badge 2 check (if all artifacts passed)

 api/v1/capability.py (requires badge_2)
 POST /capability/declaration     → upsert CapabilityDeclaration
 GET  /capability/declaration     → get current declaration
 POST /capability/document/generate → run confidence_engine + call capability_doc.py →
                                     create/update CapabilityDocument → set badge_3=True
 GET  /capability/document        → return CapabilityDocument
 PATCH /capability/document       → update ONLY narrative/positioning/focus_industries

 api/v1/execution.py (requires badge_3)
 GET  /execution/challenge        → stub: return placeholder challenge info
 POST /execution/submit           → stub: return {"status": "not_implemented"}

 api/v1/router.py — APIRouter prefix /api/v1, includes all sub-routers

 app/main.py
 - lifespan context manager: verify DB connection + Redis ping on startup
 - Include v1.router
 - CORS middleware (allow frontend URL from settings)
 - Exception handlers for HTTPException + generic 500
